# 버전 관리 가이드

> 모든 ChadApplication 프로젝트에 적용되는 버전 관리 규칙

---

## 버전 형식

```
v{MAJOR}.{MINOR}.{PATCH}

예: v0.1.0, v1.0.0, v2.3.1
```

| 자리 | 이름 | 올릴 때 | 예시 |
|------|------|---------|------|
| 첫째 (MAJOR) | 주 버전 | 디자인/기능/프레임워크 대폭 변경 | UI 리디자인, 프레임워크 교체, 기능 구조 변경 |
| 둘째 (MINOR) | 부 버전 | 새 기능 추가 | 새 화면, 새 API, 새 스캔 소스 |
| 셋째 (PATCH) | 수정 버전 | 버그 수정, 문서 업데이트, 설정 변경 | 오타 수정, README 업데이트, 포트 수정 |

---

## 언제 어떤 자리를 올리는가

### PATCH (v0.0.X) — 작은 변화

- 버그 수정
- 문서 추가/수정 (README, docs/, CHANGELOG)
- .gitignore, 설정 파일 변경
- 코드 리팩토링 (기능 변화 없음)
- 번역 추가
- 스타일/UI 미세 조정

```bash
# 예: v0.0.1 → v0.0.2
git tag v0.0.2
git push origin main --tags
```

### MINOR (v0.X.0) — 기능 추가

- 새로운 기능 구현
- 새로운 API 엔드포인트 추가
- 새로운 화면/페이지 추가
- 새로운 데이터 소스 지원 (예: R 패키지 스캔 추가)
- 성능 개선 (사용자가 체감)

```bash
# 예: v0.0.2 → v0.1.0 (PATCH 리셋)
git tag v0.1.0
git push origin main --tags
```

### MAJOR (vX.0.0) — 디자인/기능/프레임워크 변경

- UI/UX 전면 리디자인
- 프레임워크 교체 (예: React → Vue, FastAPI → Django)
- 핵심 기능 구조 변경
- DB 스키마 변경 (마이그레이션 필요)
- 인증 방식 변경
- 첫 안정 릴리스 (v0.x.x → v1.0.0)

```bash
# 예: v0.3.2 → v1.0.0 (MINOR, PATCH 리셋)
git tag v1.0.0
git push origin main --tags
```

## 실전 절차

### 1. 작업 완료 후 커밋

```bash
git add .
git commit -m "변경 내용 설명"
```

### 2. 변경 규모 판단

```
문서만 수정? 버그 수정?              → PATCH
새 기능 추가?                       → MINOR
디자인/기능구조/프레임워크 변경?      → MAJOR
```

### 3. CHANGELOG.md 업데이트

```markdown
## v0.1.0 (2026-03-17)

- 새 기능: R 패키지 스캔 추가
- 새 기능: 스캔 히스토리 + CSV/MD 다운로드
- 새 기능: 스캔 진행바 (SSE)
```

### 4. 태그 생성 + push

```bash
git tag v0.1.0
git push origin main --tags
```

### 5. 확인

```bash
git describe --tags --always
# → v0.1.0 (태그 직후)
# → v0.1.0-3-gabcdef (태그 이후 3커밋)
```

## 자동 버전 표시

웹 앱 프로젝트(template, gabrielqca, packagelist)는 `next.config.ts`에서 빌드 시 자동으로 버전을 생성합니다:

```typescript
// next.config.ts
import { execSync } from "child_process";
let gitVersion = "dev";
try {
  gitVersion = execSync("git describe --tags --always").toString().trim();
} catch {}
// → NEXT_PUBLIC_VERSION에 주입 → footer에 표시
```

태그를 생성하면 다음 빌드부터 footer에 자동 반영됩니다.

## 자주 하는 실수

| 실수 | 올바른 방법 |
|------|------------|
| 기능 추가인데 PATCH만 올림 | MINOR로 올려야 함 |
| 태그 없이 push만 함 | 반드시 `git tag` 후 `--tags`로 push |
| CHANGELOG 업데이트 안 함 | 태그 전에 CHANGELOG 먼저 작성 |
| v1.0 형식 사용 | 반드시 v1.0.0 (3자리) 사용 |
| 태그 후 수정사항 발생 | 새 태그(v0.1.1) 생성, 기존 태그 수정 금지 |
