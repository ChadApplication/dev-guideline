# 앱 개발 워크플로우

> 새로운 앱을 아이디어부터 배포까지 진행하는 7단계 프로세스

---

## 전체 흐름

```
1. 아이디어 → 2. 개발상세서 → 3. 리뷰 → 4. 셋팅 → 5. 개발 → 6. 테스트 → 7. 배포
```

---

## 1. 아이디어

**목적**: 만들고 싶은 앱의 핵심 아이디어를 기록

**산출물**: `~/Projects/_개발아이디어/{날짜}_{제목}.md`

**내용**:
- 한 줄 요약
- 문제와 해결 방향
- 대상 사용자
- 핵심 기능 (3~5개)
- 참고할 기존 프로젝트나 기술

**예시**:
```
~/Projects/_개발아이디어/20260317_PackageList.md

한 줄 요약: 컴퓨터에 설치된 패키지를 모니터링하는 로컬 대시보드
문제: 어떤 도구가 설치되어 있고 업데이트가 필요한지 파악이 어려움
핵심 기능: 스캔, 카테고리 필터, 업데이트 확인, 메모, 다국어
```

---

## 2. 개발상세서

**목적**: Claude Code와 협업하기 위한 구체적 설계 문서

**산출물**: `~/Projects/_guideline/{프로젝트}_개발상세서.md`

**템플릿 선택**:

| 상황 | 템플릿 |
|------|--------|
| 프로그래밍 처음 | `개발상세서_초간략_템플릿.md` |
| 빠른 프로토타입 | `개발상세서_간략_템플릿.md` |
| _template_latest 기반 | `개발상세서_template_latest용.md` |
| 상세 설계 필요 | `개발상세서_템플릿.md` |

**필수 기입 항목**:
- 프로젝트명, 포트 번호
- 역할 (기본 4종 유지 또는 변경)
- 추가 데이터 모델
- 추가 API
- 화면 구성
- 기능 목록 (우선순위)
- 구현 순서 (Phase)

---

## 3. 리뷰 / 피드백

**목적**: 개발 시작 전에 기술적 타당성과 설계를 검증

**방법**:
```
Claude Code에게:
"개발상세서를 읽고 리뷰하라. 기술적 문제, 누락된 기능, 개선 제안을 제시하라."
```

**체크리스트**:
- [ ] 기술 스택이 적절한가
- [ ] 데이터 모델이 기능을 지원하는가
- [ ] API 설계가 프론트엔드 요구를 충족하는가
- [ ] 구현 순서(Phase)가 논리적인가
- [ ] 보안 고려 (인증, .env, .gitignore)

**결과**: 상세서에 피드백 반영 후 "진행" 결정

---

## 4. 프로젝트 셋팅

**목적**: 실행 가능한 프로젝트 골격 생성

**절차**:
```bash
# 1. 템플릿 복사
cp -r ~/Projects/_template_latest ~/Projects/{프로젝트명}
cd ~/Projects/{프로젝트명}

# 2. run.sh 수정
#    PROJECT_NAME, DEFAULT_BACKEND_PORT, DEFAULT_FRONTEND_PORT

# 3. Git 초기화
rm -rf .git frontend/.git
git init && git branch -m main

# 4. 환경 구성
./setup.sh

# 5. 기동 확인
./run.sh start
# → http://localhost:{포트} 접속 확인
```

**완료 기준**: `./run.sh start`로 양쪽 서버 정상 기동

### 랜딩 페이지 설정

프로젝트 셋팅 시 `frontend/src/app/page.tsx`를 프로젝트에 맞게 수정:

**필수 포함 요소**:
- Hero 섹션: 프로젝트명 + 한 줄 설명 + 로그인/회원가입 버튼
- 기능 카드: 핵심 기능 3~6개 (아이콘 + 제목 + 설명)
- 워크플로우 단계: 파이프라인이 있으면 단계 시각화
- 다국어 (한/영/중) i18n 오브젝트
- Copyright footer + auto-version
- 다크 모드 지원

**참고**: gabrielqca의 `page.tsx`를 레퍼런스로 사용 (기능 카드, 파이프라인 단계, AI 페르소나 섹션 패턴)

## 5. 개발

**목적**: 상세서의 Phase 순서대로 기능 구현

**절차**:
```
Claude Code에게:
"개발상세서 Phase 1을 구현하라."
→ 완료 확인
"Phase 2를 구현하라."
→ 반복
```

**원칙**:
- Phase별로 커밋 (한번에 몰아서 하지 않기)
- 각 Phase 완료 시 브라우저에서 동작 확인
- 기능 추가 시 상세서에 체크 `[x]`
- 변경 규모에 따라 버전 태그 (PATCH/MINOR)

**커밋 메시지 형식**:
```
간결한 제목 (영문)

- 변경사항 1
- 변경사항 2

🗿 MoAI <email@mo.ai.kr>
```

## 6. 테스트 / 리뷰

**목적**: 동작 확인, 보안 점검, 코드 품질 검증

**체크리스트**:

### 동작 테스트
- [ ] 모든 페이지 접근 가능
- [ ] 핵심 기능 시나리오 통과
- [ ] 에러 상태 처리 (빈 데이터, 네트워크 오류)
- [ ] 다크 모드 확인 (해당 시)
- [ ] 다국어 확인 (해당 시)

### 보안 점검
```bash
# .gitignore 누락 확인
git ls-files | grep -iE '\.env$|dev\.db$|\.log$|node_modules|__pycache__|\.DS_Store'
# → 결과 없어야 정상
```

- [ ] `.env` 파일 미추적
- [ ] `dev.db` 미추적
- [ ] API 키 코드에 하드코딩 없음
- [ ] `*.log` 미추적

### 코드 리뷰
```
Claude Code에게:
"전체 프로젝트 리뷰하고 문제점 발견 시 수정하라."
```

## 7. 배포 / 문서화

**목적**: GitHub에 배포하고 문서를 생성

**절차**:

### GitHub 배포
```bash
# 1. 커밋
git add . && git commit -m "Initial release v0.0.1"

# 2. 태그
git tag v0.0.1

# 3. GitHub 저장소 생성 + push
gh repo create ChadApplication/{프로젝트명} --public --source=. --remote=origin --push
```

### 문서 생성
```
Claude Code에게:
"README(EN/KO/ZH), CHANGELOG, docs/ (getting-started, architecture, api) 생성하라."
```

### 산출물 체크리스트
- [ ] README.md (EN) + README_KO.md + README_ZH.md
- [ ] CHANGELOG.md
- [ ] docs/getting-started.md (+KO, ZH)
- [ ] docs/architecture.md (+KO, ZH)
- [ ] docs/api.md (+KO, ZH) — API가 있는 경우
- [ ] docs/dev-spec.md → dev-guideline 링크
- [ ] setup.sh (원커맨드 설치)
- [ ] Copyright footer + auto-version

### pm.sh 등록 확인
```bash
# ~/Projects/pm.sh에서 새 프로젝트가 보이는지 확인
cd ~/Projects && ./pm.sh
```

## 단계별 소요 시간 (참고)

| 단계 | 소요 | 비고 |
|------|------|------|
| 1. 아이디어 | 5~10분 | 자유 메모 |
| 2. 개발상세서 | 15~30분 | 템플릿 기입 |
| 3. 리뷰 | 5분 | Claude Code 리뷰 |
| 4. 셋팅 | 5분 | 템플릿 복사 + setup.sh |
| 5. 개발 | 1~수시간 | 앱 복잡도에 따라 |
| 6. 테스트 | 10~20분 | 체크리스트 기반 |
| 7. 배포 | 10~15분 | GitHub + 문서 에이전트 |

## 버전 규칙 (참고)

| 변경 | 버전 자리 | 예시 |
|------|----------|------|
| 버그 수정, 문서 | PATCH (v0.0.X) | v0.0.1 → v0.0.2 |
| 새 기능 추가 | MINOR (v0.X.0) | v0.0.2 → v0.1.0 |
| 디자인/프레임워크 변경 | MAJOR (vX.0.0) | v0.3.0 → v1.0.0 |

상세: `~/Projects/_guideline/버전_관리_가이드.md` 참조
